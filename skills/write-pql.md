"# Write a PQL Query\n\nInteractive workflow for authoring and validating PQL (Predictive Query Language) queries that translate natural-language prediction questions into executable PQL statements.\n\n---\n\n## Prerequisites\n\n- Schema knowledge: know your tables, columns, primary keys, foreign keys,\n  and time columns\n- **Read first**: `context/platform/pql-syntax.md`\n- **If debugging**: also read `context/platform/pql-errors.md`\n\n## Workflow\n\n### Step 1: Understand the Question\n\nBefore writing any PQL, decompose the natural-language question into four\ncomponents:\n\n| Component | Question to Ask | Example |\n|---|---|---|\n| **Prediction vs. Historical** | Is the answer about the future or the past? | \"Will this customer churn?\" = future prediction |\n| **Entity** | Who or what are we predicting for? | Customers, products, stores, accounts |\n| **Outcome** | What are we predicting about them? | Churn, spend, category, count of events |\n| **Time Horizon** | Over what period? | Next 30 days, next 7 days, next quarter |\n\nIf the question is purely historical or analytical (e.g., \"What was total\nrevenue last month?\"), it is a SQL question, not a PQL question. PQL is\nexclusively for predictions about future or unknown outcomes.\n\n**Decision rule**: If you can answer the question with a `SELECT` and\n`GROUP BY` on existing data, use SQL. If the answer requires forecasting\nan outcome that has not happened yet, use PQL.\n\n### Step 2: Identify the PQL Task Type\n\nMap the decomposed question to one of the supported PQL task families.\n\n| Question Pattern | Task Type | PQL Pattern |\n|---|---|---|\n| \"Will X happen in next N days?\" | Temporal binary classification | `PREDICT COUNT(target.col, 0, N, days) > 0 FOR EACH entity.pk` |\n| \"How many X in next N days?\" | Temporal regression (count) | `PREDICT COUNT(target.col, 0, N, days) FOR EACH entity.pk` |\n| \"What total X in next N days?\" | Temporal regression (sum) | `PREDICT SUM(target.col, 0, N, days) FOR EACH entity.pk` |\n| \"What average X in next N days?\" | Temporal regression (avg) | `PREDICT AVG(target.col, 0, N, days) FOR EACH entity.pk` |\n| \"What min/max X in next N days?\" | Temporal regression (min/max) | `PREDICT MIN(target.col, 0, N, days) FOR EACH entity.pk` |\n| \"What category/type is X?\" | Static classification | `PREDICT entity.category_col FOR EACH entity.pk` |\n| \"What is the value of X?\" | Static regression | `PREDICT entity.numeric_col FOR EACH entity.pk` |\n| \"What if we change Z?\" | What-if analysis | Add `ASSUMING SUM(target.col, 0, N, days) = value` |\n| \"Forecast over multiple periods\" | Multi-window forecast | Use consecutive non-overlapping windows |\n\nIf the question does not fit any pattern above, it likely cannot be\nexpressed in PQL. Common unsupported cases:\n\n- Ranking or ordering (\"Who is the top customer?\")\n- Multi-hop predictions (\"Predict X for friends of Y\")\n- Cross-entity aggregations (\"Total across all customers\")\n- Time-series decomposition (\"What is the trend?\")\n\n### Step 3: Check Schema Requirements\n\nVerify that the graph schema supports the query you plan to write.\n\n**Required checks:**\n\n1. **Entity table and PK** — `FOR EACH` column must be the primary key of\n   an entity table in the graph.\n2. **Target table and column** — Column inside `PREDICT` must exist and be\n   directly linked (1 FK hop) to the entity table.\n3. **Foreign-key path is 1-hop** — No intermediate joins allowed. Pre-join\n   in SQL if the path requires multiple hops.\n4. **Time column exists** — Temporal queries require a datetime column on\n   the target table, detected as time type (not string).\n5. **Column types match** — `SUM`/`AVG`/`MIN`/`MAX` require numeric\n   columns. `COUNT` works on any type.\n\nIf any check fails, fix the schema or graph before proceeding.\n\n### Step 4: Write the Query\n\nUse the canonical PQL template:\n\n```\nPREDICT <target>\nFOR EACH <entity_table>.<primary_key>\n[WHERE <filter_condition>]\n[ASSUMING <what_if_condition>]\n```\n\n**Target types:**\n\n| Type | Syntax | Example |\n|---|---|---|\n| Column (static) | `table.column` | `PREDICT users.segment FOR EACH users.user_id` |\n| Aggregation (temporal) | `AGG(table.column, start, end, unit)` | `PREDICT SUM(orders.amount, 0, 30, days) FOR EACH users.user_id` |\n| Binary condition | `AGG(...) > threshold` | `PREDICT COUNT(orders.order_id, 0, 30, days) > 0 FOR EACH users.user_id` |\n\n**Time window semantics:**\n\n- `(table.col, start, end, unit)` — offsets from anchor time into future.\n- Both must be non-negative integers, with `start < end`.\n- Units: `days`, `hours`, `minutes`, `months` (no `weeks`, `seconds`, or `years` — see PQL grammar).\n\n**WHERE clause (entity filter)** — filters which entities get scored:\n\n```\nPREDICT SUM(orders.amount, 0, 30, days) FOR EACH users.user_id WHERE users.status = 'active'\n```\n\n**ASSUMING clause (what-if)** — hypothetical scenario, must use\ntemporal aggregation with non-negative time window:\n\n```\nPREDICT SUM(orders.amount, 0, 30, days) FOR EACH users.user_id ASSUMING SUM(marketing.spend, 0, 30, days) = 5000\n```\n\n**WHERE inside aggregation (event filter)** — filters which rows count\ntoward the aggregation:\n\n```\nPREDICT SUM(orders.amount, 0, 30, days) WHERE orders.category = 'electronics' FOR EACH users.user_id\n```\n\n### Step 5: Pre-flight Checklist\n\nRun through every constraint before executing. A single violation will\ncause the query to fail.\n\n- [ ] **Time window non-negative** — Both `start` and `end` are >= 0, and\n  `start` < `end`.\n- [ ] **Aggregation matches column type** — `SUM`/`AVG`/`MIN`/`MAX` on\n  numeric columns only. `COUNT` on any column.\n- [ ] **ASSUMING uses temporal aggregation** — The ASSUMING clause must\n  contain a temporal aggregation (not a static column reference), and its\n  time window must be non-negative (both start and end >= 0).\n- [ ] **No nested aggregations** — `SUM(COUNT(...))` is not valid PQL.\n  Each query has exactly one aggregation level.\n- [ ] **FK path is direct (1-hop)** — The entity table and target table\n  are connected by a single foreign key. No intermediate joins.\n- [ ] **Entity column is a primary key** — The column after `FOR EACH`\n  must be the PK of its table.\n- [ ] **No explicit entity IDs with `FOR EACH`** — `FOR EACH table.pk`\n  means \"all entities\". Do not combine with `= 'id'` or `IN (...)`.\n  Pass specific entities via `model.predict(query, indices=[...])` instead.\n  Use `FOR table.pk = 'id'` only for single-entity queries without `EACH`.\n- [ ] **Table and column names are exact** — PQL is case-sensitive to the\n  graph schema. Verify spelling against `graph.print_metadata()`.\n- [ ] **No unsupported operators** — PQL supports `>`, `<`, `>=`, `<=`,\n  `=`, `!=` for binary conditions. No `BETWEEN`, `IN`, `LIKE`.\n- [ ] **Single entity table** — You cannot predict `FOR EACH` across\n  multiple tables simultaneously.\n\n### Step 6: Validate\n\nChoose the validation method that matches your environment.\n\n**In the RFM SDK (Python):**\n\n```python\nimport kumoai.experimental.rfm as rfm\n\nmodel = rfm.KumoRFM(graph)\n# predict() will validate the query before execution.\n# Invalid queries raise QueryValidationError with details.\n# indices is REQUIRED for FOR EACH queries.\nentity_ids = graph[\"entity_table\"].df[\"pk_column\"].tolist()\npred_df = model.predict(query, indices=entity_ids, run_mode=\"fast\")\n```\n\n**Using the PredictiveQuery object directly:**\n\n```python\npquery = kumoai.PredictiveQuery(graph, query_string)\npquery.validate(verbose=True)\n```\n\nIf validation fails, read the error message carefully — it usually\nidentifies the exact constraint that was violated. Proceed to Step 7.\n\n### Step 7: Iterate If Needed\n\nCommon fixes for validation failures:\n\n| Failure | Before | After |\n|---|---|---|\n| Wrong agg type | `SUM(orders.status, ...)` (categorical col) | `COUNT(orders.status, ...)` |\n| Multi-hop FK | `users -> orders -> returns` (2 hops) | Pre-join in SQL to create direct link |\n| ASSUMING without agg | `ASSUMING users.segment = 'premium'` | `ASSUMING SUM(spend, 0, 30, days) = 5000` |\n| Window reversed | `SUM(orders.amount, 30, 0, days)` | `SUM(orders.amount, 0, 30, days)` |\n\n**Consecutive-window forecast** — use non-overlapping windows:\n\n```\nPREDICT SUM(orders.amount, 0, 7, days) FOR EACH users.user_id\nPREDICT SUM(orders.amount, 7, 14, days) FOR EACH users.user_id\nPREDICT SUM(orders.amount, 14, 21, days) FOR EACH users.user_id\n```\n\n## Quick Reference: PQL Syntax\n\n**Aggregation functions:**\n\n| Function | Description | Column Type |\n|---|---|---|\n| `COUNT` | Number of rows | Any |\n| `SUM` | Sum of values | Numeric |\n| `AVG` | Mean of values | Numeric |\n| `MIN` | Minimum value | Numeric |\n| `MAX` | Maximum value | Numeric |\n\n**Time units:** `days`, `hours`, `minutes`, `months`\n\n**Comparison operators:** `>`, `<`, `>=`, `<=`, `=`, `!=`\n\n**Full query template:**\n\n```\nPREDICT AGG(table.column, start_offset, end_offset, time_unit)\n  [WHERE event_filter]\n  [> | < | >= | <= | = | != threshold]\nFOR EACH entity_table.primary_key\n  [WHERE entity_filter]\n  [ASSUMING column = value | AGG(table.column, start, end, time_unit) op value]\n```\n\n## Quick Reference: Task Type Mapping\n\n| NL Pattern | PQL Pattern | Task Type |\n|---|---|---|\n| \"Will customer churn?\" | `COUNT(orders.id, 0, 30, days) > 0` | Temporal binary |\n| \"How much will they spend?\" | `SUM(orders.amount, 0, 30, days)` | Temporal regression |\n| \"How many orders next week?\" | `COUNT(orders.id, 0, 7, days)` | Temporal regression |\n| \"What segment is this user?\" | `users.segment` | Static classification |\n| \"What if we increase budget?\" | `SUM(...) ASSUMING SUM(spend, 0, 30, days) = X` | What-if |\n| \"Predict daily revenue 4 weeks\" | Multiple `SUM(rev, N, N+7, days)` | Multi-window forecast |\n\n## Common Errors\n\n| Error | Cause | Fix |\n|---|---|---|\n| `column not found: X.Y` | Table or column name does not exist in graph | Check `graph.print_metadata()` for exact names |\n| `multi-hop FK path` | Entity and target separated by 2+ joins | Flatten with SQL join or restructure graph |\n| `invalid aggregation for column type` | `SUM`/`AVG` on non-numeric column | Switch to `COUNT` or use a numeric column |\n| `nested aggregation` | `SUM(COUNT(...))` or similar nesting | Rewrite as a single aggregation level |\n| `start >= end in time window` | Time window bounds inverted or equal | Ensure `start < end`, both non-negative |\n| `FOR EACH column is not a primary key` | Predicting for a non-PK column | Use the actual PK of the entity table |\n| `ASSUMING requires temporal aggregation` | Static column in ASSUMING clause | Wrap in temporal aggregation with non-negative window |\n| `unsupported time unit` | Typo in unit (e.g., `day` instead of `days`) | Use: `days`, `hours`, `minutes`, `months` |\n| `query is not a prediction` | Question is historical/analytical | Use SQL instead of PQL |\n| `no time column on target table` | Temporal query but table lacks datetime | Add time column to table or fix graph metadata |\n\n## Checklist\n\n- [ ] Question decomposed: entity, outcome, time horizon identified\n- [ ] Confirmed the question requires prediction (not historical SQL)\n- [ ] PQL task type selected from mapping table\n- [ ] Schema requirements verified: PK, FK path, time column, column types\n- [ ] Query written following canonical template\n- [ ] Pre-flight checklist passed (all 9 constraints)\n- [ ] Query validated via SDK, script, or PredictiveQuery object\n- [ ] Validation errors resolved (if any)\n- [ ] Final query reviewed for correctness and completeness\n- [ ] Query documented with the original natural-language question\n"
